# Data Modeling

## Critical Platform Requirements

**Primary Key Rule:**
Every record type MUST have exactly ONE primary key:
- Field name: `id`
- Field type: INTEGER
- Auto-generated by Appian

Never use composite primary keys — Appian does not support them.
Never name PK fields `case_id`, `customer_id`, etc. — must be `id`.

Use `id` + UNIQUE constraint pattern for logical keys:
```
ORDER_ITEM:
├─ id (INTEGER PK)                    ← Single surrogate key
├─ orderId (INTEGER FK)
├─ lineNumber (INTEGER)
└─ UNIQUE constraint: (orderId, lineNumber)
```

## Naming Conventions

### Database Table Names

- Format: `[PREFIX]_[ENTITY]` in UPPER_SNAKE_CASE
- Examples: `CM_CASE`, `CM_STATUS`, `CM_PRIORITY`
- Singular nouns: `CM_CASE` not `CM_CASES`
- Application prefix is mandatory for all tables
- Keep table names under 30 characters for Oracle compatibility. If over 30, remove context words (NETWORK, SYSTEM, RECORD, DATA) first.
- Status tables must follow `[ENTITY]_STATUS` format — never use generic STATUS alone
- Use entity-scoped names unless a reference table is shared by 3+ entities from different functional areas

### Database Column Names

- Format: UPPER_SNAKE_CASE
- Examples: `STATUS_ID`, `CREATED_BY`, `UPDATED_AT`, `IS_ACTIVE`
- Primary key: `ID`
- Foreign keys: `[REFERENCED_TABLE]_ID`
- Boolean fields: `IS_[CONDITION]`
- Date fields: `[CONTEXT]_DATE` or `[CONTEXT]_AT` for DATETIME

### Record Type Names

- Format: "[Prefix] Entity Name" in Title Case
- Examples: "CM Case", "CM Status", "CM Priority"
- Plural: "CM Cases", "CM Statuses", "CM Priorities"

### Record Type Field Names

- Format: camelCase
- Examples: `statusId`, `createdBy`, `updatedAt`, `isActive`
- Primary key: `id`
- Foreign keys: `[referencedEntity]Id`
- Boolean fields: `is[Condition]`
- Date fields: `[context]Date` for DATE, `[context]At` for DATETIME
- USER fields: `createdBy`, `modifiedBy`, `assignedTo` — do NOT use `Username` suffix

Appian auto-generates database column names from record type field names (camelCase → UPPER_SNAKE_CASE). You rarely need to set `sourceFieldName` explicitly.

## Normalization Principles

Design for 3rd Normal Form (3NF):

1. **Repeating groups** → Separate tables with FK relationships
2. **Categories/classifications** → Reference tables (not TEXT fields)
3. **Multi-valued attributes** → Related tables or junction tables
4. **Derived/calculated values** → Computed in queries, not stored
5. **Attributes depending on non-key fields** → Move to separate table

**Normalize When:**
- Enumerated values with 3+ options → reference table (MANDATORY)
- Multi-valued attributes → junction table
- Repeating groups → child entity with FK to parent
- Fields depending on non-key attributes → separate table

**Keep Denormalized When:**
- Exactly 2 universal binary values (Yes/No, True/False) → BOOLEAN field
- One-to-one with 3 or fewer fields → inline on parent
- Requirements say "stored with" or "stored on" parent → fields on parent

### The 1:1 Decision Tree

When you encounter a one-to-one relationship:

- **1-3 fields**: keep inline on the parent entity
- **4+ fields**: consider a separate entity (especially for sensitive data separation, optional attribute groups, or performance optimization)
- **Separate entity requires**: FK field with `isUnique: true` to enforce the 1:1 constraint, and bidirectional one-to-one relationships declared on both sides

## Reference Tables

Reference tables are small, static controlled vocabularies.

**5-Criteria Test (ALL must be true):**
1. Small, static dataset (< 50 records)
2. Controlled vocabulary structure
3. Lookup/validation purpose (dropdowns, filters)
4. No temporal/transactional attributes
5. Minimal relationships (only one-to-many outward)

**Mandatory Structure (4 fields in this order):**
```
[TABLE_NAME]:
├─ id (INTEGER PK)
├─ label (TEXT, required)                    ← Display value
├─ sortOrder (INTEGER, required)             ← Display order (1, 2, 3...)
└─ isActive (BOOLEAN, required, default: true) ← Soft delete flag
```

**Why isActive matters:**
- Deprecate values without breaking existing records
- Hide inactive values from dropdowns while preserving data
- Enable business rule changes without data migration

**Example:**
```
CM_STATUS:
├─ id (INTEGER PK)
├─ label (TEXT)
├─ sortOrder (INTEGER)
└─ isActive (BOOLEAN, default: true)

Populate with:
- Created (sortOrder: 1)
- Assigned (sortOrder: 2)
- Approved (sortOrder: 3)
- Closed (sortOrder: 4)
```

### Quick Checks: Entity vs Reference Table

- Does it grow with user actions? → Entity
- Does it have workflow/lifecycle fields? → Entity
- Does it have FKs to entity tables? → Entity
- Is it a small, static list that populates dropdowns? → Reference table
- Does its name end in _TYPE, _STATUS, _CATEGORY, _PRIORITY? → Likely reference table

### Binary Values (2 Options)

For universal binary concepts (Yes/No, True/False, Active/Inactive), use BOOLEAN — do not create a reference table. For domain-specific pairs (Approved/Rejected, Form/Request), create a reference table even though there are only 2 values.

## Entity Structure

**Standard Field Order:**
1. `id` (INTEGER PK)
2. Business fields from requirements
3. Foreign keys to reference tables and related entities
4. USER reference fields: `assignedTo`, `approvedBy`, etc.
5. Audit fields (on primary entities only)

**Entity Size Guidelines:**
- Primary entities: 10-30 fields typically
- If approaching 40+ fields, consider splitting (address, payment, custom attributes)
- Reference tables: Exactly 4 fields (id, label, sortOrder, isActive)
- Junction tables: 3-7 fields (id + 2 FKs + 0-4 metadata)

## Audit Fields (Primary Entities ONLY)

Include on primary entities (CASE, ORDER, REQUEST):

```
├─ createdAt (DATETIME, required)
├─ createdBy (USER, required → SYSTEM_RECORD_TYPE_USER)
├─ updatedAt (DATETIME, optional)
└─ modifiedBy (USER, optional → SYSTEM_RECORD_TYPE_USER)
```

Never add audit fields to:
- Reference tables (static data)
- Junction tables (only FKs)
- Event/history tables (already tracking changes)

## Soft Delete Pattern (CONDITIONAL)

Only include when requirements mention keeping history of deleted items, ability to restore, or audit trail for deletions.

```
├─ isDeleted (BOOLEAN, default=false, required)
├─ deletedAt (DATETIME, optional)
└─ deletedBy (USER, optional)
```

## Relationships

**Bidirectional Requirement (MANDATORY):**
Every FK relationship needs TWO explicit relationships — one in each direction.

**One-to-Many:**
```
CASE → STATUS:
1. CASE.statusId → STATUS.id (MANY_TO_ONE) — "Get status for case"
2. STATUS.id → CASE.statusId (ONE_TO_MANY) — "Get all cases with status"
```

**Many-to-Many via Junction:**
```
STUDENT_COURSE:
├─ id (INTEGER PK)
├─ studentId (INTEGER FK → STUDENT)
├─ courseId (INTEGER FK → COURSE)
└─ UNIQUE constraint: (studentId, courseId)

Four relationships required:
1. STUDENT_COURSE.studentId → STUDENT.id (MANY_TO_ONE)
2. STUDENT.id → STUDENT_COURSE.studentId (ONE_TO_MANY)
3. STUDENT_COURSE.courseId → COURSE.id (MANY_TO_ONE)
4. COURSE.id → STUDENT_COURSE.courseId (ONE_TO_MANY)
```

**Self-Referential:**
```
CASE:
├─ id (INTEGER PK)
├─ parentCaseId (INTEGER FK → CASE, optional)
└─ Relationships: parentCase (MANY_TO_ONE), childCases (ONE_TO_MANY)
```

Only add when requirements explicitly mention hierarchy. Never speculate.

### USER Field Relationships

Every USER field must have a MANY_TO_ONE relationship to SYSTEM_RECORD_TYPE_USER:
- System UUID: `SYSTEM_RECORD_TYPE_USER`
- User PK UUID: `SYSTEM_RECORD_TYPE_USER_FIELD_username`
- Relationship name: derive from field name (e.g., `createdBy` → `createdByUser`)

### Relationship Naming

Derive algorithmically from field names:
- **MANY_TO_ONE**: remove `Id` suffix (`statusId` → `status`, `customerId` → `customer`)
- **ONE_TO_MANY to entities**: pluralize target entity in camelCase (`CASE_NOTE` → `caseNotes`)
- **ONE_TO_MANY to junction**: pluralize the other entity (`CASE → CASE_TAG → TAG` = `tags`)
- **Self-referential inverse**: semantic plural (`parentCaseId` inverse → `childCases`)

## Document Handling

For entities requiring document attachments:

**Single document per record:** Add a DOCUMENT field directly to the entity.

**Multiple documents per record:** Create a separate `[ENTITY]_DOCUMENT` record type:
- `id` (INTEGER PK)
- `[entity]Id` (INTEGER FK → parent entity)
- `document` (DOCUMENT)
- ONE_TO_MANY relationship from parent entity

The DOCUMENT field type references Appian's built-in document system. Use `SYSTEM_RECORD_TYPE_DOCUMENT` (key field: `SYSTEM_RECORD_TYPE_DOCUMENT_FIELD_id`) when creating relationships to the system document record type.

Use entity-specific naming for document tables (e.g., `CM_CASE_DOCUMENT`, `CM_REQUEST_DOCUMENT`).

## Unique Constraints

**Single-Field (Enforced at Database):**
Use `isUnique: true` on the field — email, employee number, license plate.

**Compound (Documented Only — Application Logic):**
Document in implementation notes: "UNIQUE constraint: (orderId, lineNumber)"
- Combinations that must be unique together
- Cannot be enforced at database level in Appian
- Must be handled in application logic

## External System IDs (CONDITIONAL)

Only when requirements mention integration/sync:
```
CUSTOMER:
├─ id (INTEGER PK)                    ← Appian internal ID
├─ salesforceId (TEXT, UNIQUE)        ← External system ID
├─ lastSyncedAt (DATETIME, optional)
```

Keep `id` as PK for internal relationships. Never use external IDs as FKs.

## Validation Checklist

**Structural:**
- [ ] Every table has single-column INTEGER PK named `id`
- [ ] All junction tables have exactly 2 FKs + UNIQUE constraint
- [ ] All FK fields have corresponding relationships declared
- [ ] All one-to-many have corresponding many-to-one on other side
- [ ] No orphaned tables without relationships

**Naming:**
- [ ] Table names: UPPER_SNAKE_CASE with application prefix, singular
- [ ] Record type names: Title Case with application prefix
- [ ] Field names: camelCase
- [ ] PK field: `id` (not case_id, customer_id)
- [ ] FK fields: `[referencedEntity]Id`

**Consistency:**
- [ ] No duplicate field names within any record type
- [ ] All relationship names unique within a record type
- [ ] Every USER field has MANY_TO_ONE to SYSTEM_RECORD_TYPE_USER
- [ ] Reference tables have 4 fields: id, label, sortOrder, isActive
- [ ] Primary entities have audit fields (createdAt, createdBy, updatedAt, modifiedBy)
- [ ] Junction tables have NO audit fields

**Platform:**
- [ ] Only valid Appian field types used
- [ ] No user/role entities (use USER fields instead)
- [ ] All tables conform to 3NF minimum

## Common Pitfalls

- Using `case_id`, `customer_id` for PK — must be `id`
- Omitting application prefix from table/record type names
- Using composite primary keys — Appian doesn't support them
- Creating user/employee entities — use USER fields instead
- Adding audit fields to reference/junction tables — only on primary entities
- Using generic STATUS table — always scope: CM_STATUS, CM_PRIORITY
- Forgetting bidirectional relationships — both sides required
- Adding relationship without FK field (or vice versa) — need both
- Using `allowed_values` for 3+ options — create reference table
- Speculatively adding self-referential — only when explicitly required
- Using `createdByUsername` instead of `createdBy` — USER fields don't use Username suffix
- Fabricating usernames in sample data — query SYSTEM_RECORD_TYPE_USER for real usernames
